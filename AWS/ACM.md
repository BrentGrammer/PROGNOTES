## HTTPS/SSL

- Uses SSL or TLS protocols to create a secure tunnel that normal HTTP can be transfered through

## Certificates

- ACM can generate or import certificates
  - generated certs are automatically renewed on your behalf
  - **Imported certs are not renewed automatically** and you are responsible for renewing them (need to renew them with the external source and import them again)
- Allows servers to authenticate their identity using SSL/TLS with a certificate
  - Certificates can be digitally signed by a known Certificate Authority trusted by the Web Client
  - Chain of trust:
    - Web Client trusts the CA > You trust the certificate it signs, so you trust the site itself
    - If a site claims to be Netflix.com and has the Netflix.com certificate signed by a trusted CA, then it is almost gauranteed to be Netflix.com and not a spoof
  - For a web site to be considered secure:
    - Picks a domain name
    - Generates a certificate or has one generated for it
    - Has that certificate signed by a trusted CA and uses that to prove its identity
    - **The DNS name and the certificate are tied together**
- ACM can fuction as a PUBLIC Certificate Authority or a Private CA
- Browsers trust ACM as a Certificate Authority - if they find certificates generated by ACM which are deployued to edge locations or load balancers (AWS Albs) they access, they will be trusted.

### Public Certificates

- Public certs are signed and recognized at large by browsers
- Browsers trust a list of providers (provided by the Operating System vendor), which can trust other providers (chain of trust)

### Private Certificates

- Private certs are used by large corporations for internal use. You configure clients so that they trust the private Certificate Authority
- (adding this trust into client laptop and desktop builds manually, or via a policy to configure this trust)

### Supported Services for ACM

- ACM can only generate and deploy certs to supported AWS services and not all services are supported.
- ONLY CloudFront and Application Load Balancers -- **DOES NOT SUPPORT EC2**
  - The point of ACM is to deploy securely and encrypted the certificates, and EC2 is self managed which means you can always access it as a root user.
- **Regional Service** (isolated in region, i.e. us-east-1 and another in us-west-2, etc.)
  - If you generate a Certificate in one region, it cannot leave that region and are locked to that region.
  - Example: if you want to use a cert with a ALB in us-east-1, the cert must be in ACM in us-east-1
  - **EXCEPTION**: For CloudFront, even though it is a global service, you view it as running in `us-east-1` from the ACM perspective.
    - _Always use `us-east-1` region for CloudFront Certificates in ACM_
    - When the certificate is linked to the CloudFront Distribution, the Distribution can deploy it to the edge locations across all regions
- Note: S3 Does not use ACM for certificates
